What Microsoft Fabric Actually Is (And What It Isn't)

Microsoft Fabric is not a single product. It is a unified SaaS platform built on OneLake — a single logical data store built on Azure Data Lake Storage Gen2 with Delta Parquet as the native format. Sitting on top of OneLake are six workloads: Data Engineering (Spark notebooks and Lakehouses), Data Factory (pipelines), Data Warehouse (T-SQL), Real-Time Intelligence (Eventstream, KQL, Data Activator), Data Science (ML experiments), and Power BI.

The unifying principle is that all six workloads read from and write to the same OneLake storage. There is no data movement between workloads — a Spark notebook and a Power BI Direct Lake semantic model are reading the same Delta files. This is fundamentally different from Azure Synapse, where data moved between Synapse Spark, Synapse SQL, and Power BI datasets via separate connections and refresh processes.

What Fabric is not: it is not a replacement for all Azure data services. Azure Data Factory, Azure Databricks, Azure SQL Database, and Azure Event Hubs all have use cases that remain valid. For most Australian organisations, Fabric is not a rip-and-replace — it is the primary analytics platform, with Azure services retained where Fabric does not have a native equivalent.

The Case for Migrating from Azure Synapse

Microsoft has been explicit: Azure Synapse Analytics is in sustaining mode. No significant new capabilities are being shipped; the investment is in Fabric. For Australian organisations running Synapse dedicated pools, the strategic question is not whether to migrate — it is when and how.

The financial case is straightforward for most organisations. A Synapse dedicated pool running at DW1000c incurs compute cost whether queries are running or not. Fabric F-SKUs use Compute Units (CUs) that can be paused and scaled, and the Direct Lake semantic model eliminates the Power BI dataset refresh overhead that was a significant compute driver in many Synapse environments.

The architectural case is equally clear. Synapse workspaces required multiple linked services, integration runtimes, and pipeline activities to move data between its own components. Fabric's unified storage eliminates most of that movement. An architecture that required six services in Synapse often requires two or three in Fabric.

Medallion Architecture on Microsoft Fabric: What it Looks Like in Production

The Bronze / Silver / Gold medallion pattern is the dominant architecture for Microsoft Fabric implementation in Australia. In production, it looks like this:

Bronze Lakehouse holds raw source data exactly as it arrived — no transformations, no schema changes, no validation. Fabric Pipelines (or ADF for complex orchestration) load Bronze via REST, SFTP, SAP ODP, SharePoint, or direct database connectors. Bronze is append-only and partitioned by ingestion date. It is the archive of record.

Silver Lakehouse holds conformed, typed, and validated data. PySpark notebooks run transformations from Bronze to Silver on a schedule or via event trigger. Schema validation, null handling, deduplication, and business key standardisation happen in Silver. Silver is the layer that data engineering tests are written against.

Gold Lakehouse or Fabric Warehouse holds aggregated, metric-calculated data optimised for consumption. Gold can be either Delta tables in a Lakehouse (for Direct Lake semantic models) or a Fabric Warehouse (for T-SQL consumers). For most Australian organisations, Direct Lake from a Gold Lakehouse provides the best balance of query performance and data freshness.

Direct Lake: The Feature That Changes the Power BI Model

Direct Lake is the architectural shift that makes Fabric the default choice for Power BI-heavy organisations. In the traditional Power BI model, datasets are either Import (copy of data, fast queries, refresh required) or DirectQuery (live queries, slower, no refresh). Direct Lake reads Delta Parquet files in OneLake directly into the Vertipaq engine — achieving Import query speeds without maintaining a copy and without requiring scheduled refresh.

The practical implication: Power BI reports that previously had overnight or hourly refresh schedules show live data without any refresh configuration. For Australian financial services organisations running daily regulatory reporting, this alone is a significant operational change — reports that were based on yesterday's data are now based on today's.

Direct Lake has constraints: it requires the semantic model to be in a Fabric workspace with a Fabric capacity (not Power BI Premium). And not all DAX expressions are fully supported in Direct Lake fallback scenarios. These constraints are well-understood and manageable for a prepared Microsoft Fabric consultant — they are a surprise for teams who assumed Direct Lake would be a drop-in replacement for their existing datasets.

APRA CPS 234 and Microsoft Fabric: What Regulated Organisations Need to Know

Australian financial services organisations operating under APRA's CPS 234 (Information Security) have specific obligations around data security, access control, and incident reporting that apply to Fabric tenancy design. The key considerations are:

Fabric tenant isolation: OneLake is a tenant-level resource. A poorly-configured Fabric tenancy can expose workspaces across different business units or external contractors. Role assignments at workspace level must be governed centrally — not delegated to individual workspace owners without controls.

Private Link and network security: Fabric supports Private Link for network isolation. For APRA-regulated entities with strict network segmentation requirements, Private Link must be configured before production data is ingested. Retrofitting Private Link after go-live requires a maintenance window and can cause pipeline disruption.

Data lineage and audit trails: APRA CPS 234 expects demonstrable control over information assets. Microsoft Purview integrated with Fabric provides automated data lineage from source to semantic model. For regulated organisations, this is not optional — it is the evidence that satisfies auditor requests.

The Migration Sequence That Works

Every Microsoft Fabric migration ProcessBI has run in Australia follows the same five-stage sequence. Deviating from it has, in every case, added time rather than saved it.

Stage 1 — Tenancy audit. Document existing data estate: pipeline inventory, semantic model count, report count, user access model, and compute costs. This is the baseline against which migration ROI will be measured.

Stage 2 — Fabric tenancy design. Define workspace topology, F-SKU sizing, security model (Entra ID groups, RLS, OLS, Private Link), and OneLake folder structure. Get signoff before building anything.

Stage 3 — Bronze build. Migrate or rebuild ingestion pipelines to load raw data into Bronze Lakehouse. Validate row counts against source.

Stage 4 — Silver/Gold build and semantic model cutover. Build Silver transformations in PySpark. Build Gold aggregations. Migrate Power BI datasets to Direct Lake semantic models. Validate DAX results against legacy reports before switching users.

Stage 5 — Parallel run and cutover. Run Fabric and legacy platform simultaneously for 2–4 weeks. Compare outputs on the same live data. Decommission legacy services after validation passes.

The parallel run is not optional. Every migration ProcessBI has run has found at least one meaningful discrepancy during the parallel period — a transformation logic difference, a timezone handling issue, a rounding behaviour in DAX that produced a different result from the legacy calculation. Catching these during parallel is the point.
