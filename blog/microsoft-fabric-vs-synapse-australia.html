Microsoft Fabric vs Azure Synapse: The Key Differences

Azure Synapse Analytics was Microsoft's attempt to unify data warehousing, data integration, and analytics on a single platform. It succeeded partially: the dedicated SQL pool (formerly SQL DW) remained the core compute layer, pipelines were a wrapper around ADF, and Synapse Notebooks were a thin Spark runtime. The experience across these components was inconsistent and the billing model complex.

Microsoft Fabric is a different architectural philosophy. OneLake is the universal storage layer — a single Delta Parquet store shared across Lakehouses, Warehouses, KQL Databases, and Eventstreams. All Fabric workloads read from and write to the same physical storage, eliminating data copying between services. Billing is unified through capacity (F-SKUs or P-SKUs), replacing the per-service billing of Synapse.

When Should an Australian Organisation Migrate to Fabric?

Not every organisation should migrate immediately. The decision depends on where the cost-benefit falls for your specific context.

Migrate now if: your Synapse dedicated pool is your primary cost driver and you have predictable query patterns that would benefit from Direct Lake; you are running multiple Synapse workspaces that could consolidate onto a single Fabric capacity; or you have a strategic requirement for real-time analytics that Fabric Eventstream and KQL serve better than Synapse RT.

Wait if: you have significant SSIS-based ETL that has not yet been migrated to ADF; you rely on Synapse Link features that do not yet have direct Fabric equivalents; or you are in a regulated environment that requires Fabric features to reach IRAP or ISM maturity before you can adopt them.

Accelerate if: you are planning a new data platform build. Starting on Fabric today avoids building on a platform Microsoft is de-investing, and the architecture you would build on Synapse and the architecture you would build on Fabric are similar enough that the design work translates directly.

The Medallion Architecture in Microsoft Fabric

The medallion (Bronze / Silver / Gold) architecture is the standard pattern for Microsoft Fabric implementation in Australia and globally. It organises data into three logical layers within OneLake:

  • Bronze (raw): data as it arrives from source systems — no transformation, no validation. Preserves the exact state of the source at ingestion time. Enables replay from source if transformation logic changes.
  • Silver (conformed): cleaned, typed, and business-rule-validated data. Typically structured as Delta tables with consistent schemas. The Silver layer is where most data quality and governance controls apply.
  • Gold (served): aggregated, metric-calculated, and domain-aligned data optimised for consumption. Gold tables feed Direct Lake semantic models and Fabric Warehouses for SQL-based reporting.

The medallion pattern is not new — it derives from Databricks' original lakehouse architecture — but Fabric's OneLake makes it native rather than assembled from separate services. All three layers share the same physical storage; there is no data movement between layers, only in-place transformation and Delta table updates.

Direct Lake: The Core Benefit for Power BI

Direct Lake is the feature that fundamentally changes Power BI's relationship with data in Fabric. In traditional Power BI semantic models, data is either imported (copied into the model's compressed in-memory store) or queried via DirectQuery (live queries to the source at report load time). Both approaches have trade-offs: Import is fast but creates a copy that requires refresh; DirectQuery is always current but query performance depends on the source system.

Direct Lake reads Delta Parquet files in OneLake directly into the Vertipaq memory engine — without importing a copy and without issuing live queries. When data in OneLake changes, the semantic model sees the change immediately on the next query. The result is Import-equivalent query performance with DirectQuery-equivalent freshness.

The practical implication for Australian organisations migrating from Synapse: Power BI reports that were previously on Import schedules (refreshing overnight or hourly) can switch to Direct Lake and show live data without any refresh configuration. That eliminates a category of operational overhead that affects most large Power BI estates.

How a Microsoft Fabric Migration Should Be Sequenced

ProcessBI's Fabric migration consultant approach follows a five-stage sequence that keeps business reporting operational throughout:

  • Stage 1 — Tenancy audit: Document existing Synapse workspaces, pipeline dependencies, semantic model counts, and report inventory. Establish the current-state architecture baseline.
  • Stage 2 — Fabric tenancy design: Define workspace structure, capacity sizing (F-SKU selection), security model (Entra ID groups, RLS/OLS, Private Link), and the Bronze/Silver/Gold layer design.
  • Stage 3 — Bronze build: Migrate ADF pipelines or SSIS packages to Fabric Pipelines. Ingest raw data into OneLake Bronze Lakehouse. Validate row counts against Synapse source.
  • Stage 4 — Silver/Gold transformation and semantic model cutover: Build Silver transformations in PySpark notebooks. Build Gold aggregations as Delta tables. Migrate SSAS Tabular or legacy Power BI import datasets to Direct Lake semantic models. Validate DAX results against existing reports.
  • Stage 5 — Parallel run and cutover: Run Fabric and Synapse in parallel for a defined period (typically 2–4 weeks). Confirm report outputs match. Decommission Synapse workspaces and dedicated pool.

Common Migration Pitfalls

Migrating the current architecture directly instead of redesigning it. Synapse architectures were often built around the constraints of Synapse — multiple workspaces for separation, ADF pipelines to move data between services, import datasets to avoid DirectQuery latency. Those workarounds are unnecessary in Fabric. Migrating them as-is imports technical debt. The migration is an opportunity to clean the architecture, not just move it.

Under-sizing the Fabric capacity. Fabric F-SKU selection affects both query performance and pipeline throughput. Under-sizing leads to throttling during parallel runs — which makes the parallel run look like Fabric is slower than Synapse, when the problem is capacity configuration. Right-sizing requires load testing under realistic query patterns before go-live.

Not running a parallel period. The most common source of migration risk is cutting over without a period of parallel running where Fabric outputs are validated against Synapse outputs on the same live data. Every Microsoft Fabric migration project ProcessBI has run has found at least one discrepancy during parallel run that would have been invisible without it.
